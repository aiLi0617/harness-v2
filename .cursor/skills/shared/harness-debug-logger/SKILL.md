# Harness 全局调试日志

## 用途
记录 harness 工程运行时的完整轨迹，用于调试工作流执行过程。

## 触发时机
每个工作流的每个步骤执行前后自动记录：
- 工作流启动/完成
- 规则加载（列出所有激活的 .mdc）
- **资源冲突检测**（规则加载后立即执行）
- 技能调用（进入/退出）
- 子代理派发（启动/完成）
- 审查闭环（每轮结果）
- 人工检查点（暂停/恢复）

## 输出文件
`docs/artifacts/harness-debug.md`

## 日志条目格式

每条日志追加到 harness-debug.md，格式如下：

### 工作流启动
```
## [YYYY-MM-DD HH:mm:ss] 工作流启动
- **工作流**: {workflow-name}
- **任务**: {task-name}
- **触发方式**: 用户指令
```

### 规则加载
```
## [YYYY-MM-DD HH:mm:ss] 规则加载
- **已激活规则**:
  - `rules/{layer}/{name}.mdc`
  - ...
```

### 资源冲突检测

工作流启动后、第一个步骤执行前，扫描 **harness 工程自身**的激活资源检测冲突。

**扫描范围（仅限 harness 工程配置）**：

- `.cursor/rules/**/*.mdc` — 本次工作流激活的规则文件
- `.cursor/skills/**/SKILL.md` — 本次工作流激活的技能文件
- `.cursor/agents/**/*.md` — 本次工作流派发的子代理定义
- `.cursor/workflows/*.yaml` — 当前工作流编排文件

**明确不在扫描范围内**：

- ❌ 业务项目源码（`src/**/*.java`、`*.xml`、`*.sql` 等）
- ❌ 业务项目文档（`docs/` 下的 PRD、LLD、DDL 等制品）
- ❌ 业务项目配置（`pom.xml`、`application.yml` 等）

> 注：业务项目的代码/制品/配置一致性检查由对应阶段的子代理（如 `consistency-reviewer`、`schema-guard`）负责，不在 harness 启动时的资源冲突检测之列。

**检测的冲突类型**（仅针对 harness 配置之间的关系）：

| 冲突类型 | 说明 |
|---------|------|
| 职责重叠 | 两条规则 / 两个技能对同一件事都有定义 |
| 定义矛盾 | 两条规则 / 两个技能对同一件事给出相反指令 |
| 引用缺失 | 一个规则 / 技能 / 代理引用了另一个未被激活的 harness 资源 |
| 覆盖空白 | 工作流某个阶段没有任何 harness 规则 / 技能覆盖 |

```
## [YYYY-MM-DD HH:mm:ss] 资源冲突检测
- **扫描范围**: harness 配置（N 条规则 + N 个技能 + N 个子代理 + 1 个工作流 YAML）
- **发现冲突**: N 项

### 冲突 #1 — {冲突类型}
- **资源 A**: `{path}` — {定义摘要}
- **资源 B**: `{path}` — {定义摘要}
- **冲突点**: {描述}
- **建议**: {处理方式}
```

### 技能调用
```
## [YYYY-MM-DD HH:mm:ss] 技能调用 — {skill-name}
- **阶段**: {phase}
- **技能**: `{skill-path}`
- **输入**: {输入摘要}
- **输出**: {输出摘要，含产出文件路径}
- **引用文件**: {读取的文件列表}
```

### 子代理派发
```
## [YYYY-MM-DD HH:mm:ss] 子代理派发 — {agent-name}
- **阶段**: {phase}
- **子代理**: `{agent-path}`
- **模式**: 只读/读写
- **输入**: {读取制品列表}
- **输出**: {产出制品列表}
- **引用文件**: {相关文件}
```

### 审查闭环
```
## [YYYY-MM-DD HH:mm:ss] 审查闭环 — {reviewer-name}（第 N 轮）
- **阶段**: {phase}
- **审查者**: `{reviewer-path}`
- **输入**: {制品列表}
- **审查结果**: 通过/不通过
- **轮次**: N/5
- **处理**: {通过→进入下一阶段 / 回退给xxx修正}
```

### 人工检查点
```
## [YYYY-MM-DD HH:mm:ss] 人工检查点
- **阶段**: {phase}
- **原因**: {暂停原因}
- **问题**: 详见 `docs/artifacts/decision-log.md` 决策 #N
- **用户决策**: {摘要}
- **状态**: 已恢复
```

### 制品归档
```
## [YYYY-MM-DD HH:mm:ss] 制品归档
- **任务**: {task-name}
- **归档路径**: `docs/artifacts/archive/{date}-{task-name}/`
- **归档文件数**: N 个（含本日志）
- **状态**: 工作流完成
```

## 可选开关
通过 workflow YAML 中 `debug: true/false` 控制，默认启用。

## 并行场景
当使用 git worktree 并行开发时，各并行 agent 写入独立日志文件（harness-debug-worker-N.md），收尾时合并为一个。

## 关键约束
- 粒度：步骤级别，不记录代码行级细节
- 必录字段：时间戳、阶段、资源路径、输入摘要、输出摘要
- 引用文件：每条日志标注读取和产出的文件
- 归档：日志本身也是制品，随其他制品一起归入 archive/
- 与 decision-log.md 去重：人工检查点只记摘要+引用，不重复完整问答
- 与审查报告去重：审查闭环只记通过/不通过+问题数，引用报告文件
- ⚠️ 编码阶段（阶段⑦）每个子任务（T1/T2/...）完成后必须**立即追加**对应日志条目，禁止"先全部改完再一起补录"；若在编码收尾审查时发现日志缺失，必须复盘根因并在 harness-debug.md 追加一条"日志补录"条目记录此次违规，而非安静补齐
